## 3. Local development

### 3.1 Prerequisites

- Node РЅЦ 20
- pnpm or npm
- Supabase CLI (`brew install supabase/tap/supabase` or `npm i -g supabase`)
- A Supabase project (you already have **priyasapp**)

### 3.2 First-time setup

1. **Rotate the credentials you previously shared in chat** at
   `Supabase Dashboard Рєњ Settings Рєњ Database` (password) and
   `Рєњ API` (anon/publishable + service-role keys). Do this *before* anything
   else.
2. `cp .env.example .env.local` and fill in:

   ```env
   NEXT_PUBLIC_SUPABASE_URL=https://oeovzmsstlaqbfqntwjz.supabase.co
   NEXT_PUBLIC_SUPABASE_ANON_KEY=РђдrotatedРђд
   NEXT_PUBLIC_SUPABASE_PROJECT_ID=oeovzmsstlaqbfqntwjz
   SUPABASE_SERVICE_ROLE_KEY=Рђдrotated, server-onlyРђд
   NEXT_PUBLIC_APP_URL=http://localhost:3000
   ```
3. Install deps: `pnpm install` (or `npm install`).
4. Link the Supabase CLI: `supabase link --project-ref oeovzmsstlaqbfqntwjz`.
5. Apply migrations: `pnpm supabase:push`. (This runs every file in
   `supabase/migrations/` in order. They are idempotent Рђћ safe to re-run.)
6. (Optional) Load dev seed data:
   `psql "$DATABASE_URL" -f supabase/seed/seed.sql`.
7. Generate database types so `Database` in `src/types/database.ts` is real:
   `pnpm supabase:types`.
8. Run: `pnpm dev` Рєњ http://localhost:3000.

### 3.3 Creating your first user

The signup flow is closed (`auth.enable_signup = false` in `config.toml`).
Use the Supabase dashboard:

1. **Authentication Рєњ Users Рєњ Add user** (email + password).
2. **Edit user Рєњ Raw user meta data**, set:
   ```json
   {
     "org_id": "00000000-0000-0000-0000-0000000000aa",
     "full_name": "Your Name",
     "role": "admin"
   }
   ```
   The `handle_new_user()` trigger creates the matching row in `public.profiles`.
3. Sign in at `/login`.

(For a real org you'll create one organization row, then attach users to it
the same way.)

## 4. Database

### 4.1 Migrations

Every change to the schema is a new file in `supabase/migrations/`,
named `YYYYMMDD_NNNNNN_<description>.sql`. Migrations must be idempotent
(use `create Рђд if not exists`, `drop policy if exists`, etc.) so they can be
re-run during reviews.

Apply locally / to staging / to production:

```bash
pnpm supabase:push          # push pending migrations
pnpm supabase:reset         # nuke local DB + replay all migrations + seed
```

### 4.2 RLS

Every domain table is org-scoped. The pattern is:

- **read:** `org_id = current_org_id()`
- **insert / update:** `Рђд AND is_dispatcher_or_admin()`
- **delete:** `Рђд AND is_admin()`

The helpers `current_org_id()`, `current_user_role()`, `is_admin()`,
`is_dispatcher_or_admin()` live in `20260504_000001_foundation.sql`.

### 4.3 Storage

Buckets `property-photos`, `employee-docs`, `invoice-pdfs`, `chat-attachments`
follow the path convention `<org_id>/Рђд`. Storage policies enforce that prefix.

## 5. Adding a feature page

A typical end-to-end addition (e.g. *Clients list*) looks like:

1. **Validators** Рђћ write `src/lib/validators/clients.ts` (Zod).
2. **Hooks** Рђћ `src/hooks/clients/useClients.ts`,
   `useCreateClient.ts`, `useUpdateClient.ts`. Each hook wraps a TanStack Query
   call to a typed Supabase query.
3. **Server actions** Рђћ for mutations, define `'use server'` actions colocated
   under the route or in `src/lib/api/clients.ts`.
4. **Page** Рђћ replace `src/app/(dashboard)/clients/page.tsx` with the real
   component. Use the design tokens Рђћ never re-define colors. Lift repeated
   markup into `src/components/clients/`.
5. **Tests** Рђћ add Playwright/Vitest specs (CI hooks coming in Phase 2).

## 6. Adding a database table

1. New migration: `supabase/migrations/<ts>_<name>.sql`.
2. Define the table, indexes, `updated_at` trigger, soft-delete column.
3. Enable RLS and write the standard policies (see existing migrations).
4. `pnpm supabase:push && pnpm supabase:types`.
5. Build hooks + page as in ┬Д5.

## 8. Deployment (Vercel)

1. Push the repo to GitHub.
2. **Import project** in Vercel; framework = Next.js, root = `app/`.
3. Set the env vars from `.env.example` in *Project Settings Рєњ Environment Variables*
   (Production + Preview). Use the **server-only** key (`SUPABASE_SERVICE_ROLE_KEY`)
   only as an *Encrypted* env var.
4. **Domain** Рєњ assign your custom domain.
5. Add the Vercel domain to **Supabase Рєњ Authentication Рєњ URL Configuration**
   (`Site URL` and `Redirect URLs`). Without this, OAuth / email links break.
6. Deploy. Verify `/api/health` returns `{ ok: true }`.

## 10. Troubleshooting

- **`Invalid environment variables`** on boot Рђћ `src/lib/constants/env.ts`
  refused a value; fix `.env.local` per the schema.
- **Login succeeds but dashboard 500s** Рђћ most likely no `profiles` row for
  the user. Set `raw_user_meta_data.org_id` in the dashboard, then sign in
  again.
- **RLS errors (`permission denied`)** Рђћ check that
  `select public.current_org_id();` returns a UUID for the signed-in user
  in `Database Рєњ SQL Editor`. If null, the trigger didn't run.
